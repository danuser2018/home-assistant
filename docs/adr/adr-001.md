# ADR-001: Usar el Sistema de Ficheros como Bus de Mensajes entre Servicios

## Fecha
28-06-2026

## Estado
Aceptado

## Contexto
El asistente de voz requiere comunicar daemons que corren nativamente en el host Linux (como `mic-daemon` y `speaker-watchdog`) con microservicios lógicos ejecutándose dentro de contenedores aislados de Docker (`interaction-manager`, `stt-capability`, `orchestrator`, etc.). Esta comunicación debe ser fiable, desacoplada, fácil de observar y con una sobrecarga operacional mínima en entornos locales.

## Decisión
Utilizaremos el sistema de ficheros local compartiendo un directorio (`data/`) mediante volúmenes de Docker como bus de paso de mensajes. 

El flujo se coordina a través de subdirectorios específicos que representan estados del pipeline de voz:
- `data/input/`: Bandeja de entrada para archivos de audio grabados por el micrófono.
- `data/processing/`: Estado transitorio durante el procesamiento de la petición.
- `data/output/`: Archivos de voz sintetizados listos para su reproducción.
- `data/error/`: Depósito de depuración para archivos que fallaron en el pipeline.

Los servicios reaccionarán a los cambios en estos directorios mediante observadores de eventos del sistema de archivos (`inotify` o la librería `watchdog` en Python) o mediante polling de baja frecuencia.

## Alternativas consideradas
- **Message Broker Centralizado (RabbitMQ / MQTT / Redis):** Descartado por añadir complejidad y dependencias de infraestructura excesivas para el objetivo local del asistente.
- **HTTP síncrono de extremo a extremo:** Descartado porque acopla los daemons del host al ciclo de vida de la red de Docker y dificulta el manejo de colas de reproducción o audio simultáneo.
- **Sockets Unix o TCP directos:** Descartado por la complejidad de configurar permisos y mapear sockets dinámicos dentro de contenedores de Docker.

## Consecuencias
+ **Simplicidad extrema:** Cero dependencias o servicios adicionales para la mensajería.
+ **Observabilidad inmediata:** Cualquier desarrollador o script puede inspeccionar la cola de procesamiento mediante un simple `ls` o visualizando el contenido de las carpetas.
+ **Depuración trivial:** Los fallos se auto-documentan moviendo el mensaje original a `data/error/` para su inspección posterior.
+ **Desacoplamiento total:** Los servicios no necesitan conocer las direcciones IP, puertos o tecnologías de los otros; solo necesitan leer/escribir en el filesystem.
- **Uso de I/O de disco:** Genera múltiples escrituras, lo cual se mitiga almacenando carpetas temporales de control en `/tmp` (RAM-backed TMPFS).
- **Acoplamiento físico:** Requiere que los servicios compartan acceso físico al mismo disco o red local, dificultando la distribución en múltiples hosts sin montar sistemas de ficheros de red compartidos.
