# ADR-010: Aislamiento de Variables de Entorno por Servicio

## Fecha
02-07-2026

## Estado
Aceptado

## Contexto
Históricamente, el ecosistema Nova-2 utilizaba un único archivo de variables de entorno compartido `config/assistant.env` que se cargaba en múltiples servicios dentro del archivo `docker-compose.yml` (STT, TTS, Orchestrator, System Service, Mail Watchdog e Identity Service).

Esta configuración compartida violaba el principio de mínimo privilegio y el aislamiento de dominios, ya que todos los contenedores tenían visibilidad total sobre las variables de los demás servicios. Por ejemplo, el contenedor `mail-watchdog` tenía acceso a datos de identidad del usuario como `USER_NAME` y `USER_EMAIL` (propios de `identity-service`), mientras que otros contenedores tenían acceso a secretos SMTP como `SMTP_PASSWORD` (propios de `mail-watchdog`).

## Decisión
Para mitigar los riesgos de seguridad y reforzar el aislamiento de dominios, se decide:
1. **Eliminar el archivo unificado**: Suprimir definitivamente el archivo global `config/assistant.env`.
2. **Crear configuraciones específicas**: Introducir archivos `.env` independientes para cada servicio que requiera variables configurables por el usuario:
   - `config/stt-capability.env` (Whisper model, device)
   - `config/tts-capability.env` (Piper voice name, URL)
   - `config/system-service.env` (Nova assistant identity info)
   - `config/orchestrator.env` (LogLevel)
   - `config/mail-watchdog.env` (SMTP credentials)
   - `config/identity-service.env` (User identity data)
   - `config/interaction-manager.env` (Poll interval, language, log level, timeouts)
3. **Mapeo en Compose**: Actualizar la directiva `env_file` de cada contenedor en `docker-compose.yml` para apuntar a su archivo específico.
4. **Variables de Infraestructura Inline**: Mantener las variables que representan topología física o infraestructura del entorno Docker (tales como `SYSTEM_SERVICE_BASE_URL`, `MAIL_PENDING_DIR` e `IDENTITY_SERVICE_BASE_URL`) declaradas de forma inline bajo la sección `environment:` en `docker-compose.yml`, ya que no constituyen parametrizaciones del usuario final.
5. **Scripts del Host**: Modificar `scripts/install.sh` y `scripts/update.sh` para extraer las variables del modelo de voz desde el archivo específico `config/tts-capability.env`.

## Alternativas consideradas
- **Mantener la configuración unificada**: Rechazada debido al acoplamiento de seguridad y violación del principio de mínimo privilegio.
- **Utilizar variables puramente inline en Docker Compose**: Rechazada para evitar obligar al usuario final a editar el archivo de definición del motor de contenedores (`docker-compose.yml`) para realizar configuraciones básicas (como credenciales SMTP o datos del usuario).

## Consecuencias
+ **Seguridad Reforzada**: Se limita el radio de exposición de secretos del sistema. Las credenciales SMTP solo son visibles dentro del contenedor de envío de correos.
+ **Aislamiento de Dominios**: Cada microservicio consume exclusivamente sus parámetros de negocio asociados.
+ **Mantenibilidad**: Los cambios en la configuración de un componente no impactan ni exponen información de otros.
- **Complejidad de Ficheros**: Aumenta el número de archivos `.env` gestionados en la carpeta `config/`, aunque esto se alinea con las mejores prácticas de la arquitectura de microservicios.
