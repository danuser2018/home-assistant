# ADR-006: Cola de Mensajería Asíncrona basada en Ficheros JSON

## Fecha
28-06-2026

## Estado
Aceptado

## Contexto
El asistente de voz permite realizar acciones secundarias externas que no pertenecen al pipeline de procesamiento de voz en tiempo real (por ejemplo, notificar información de capacidades del sistema al correo electrónico del usuario mediante el plugin de `capabilities` del `orchestrator`). Si estas operaciones secundarias, que dependen de servicios de red externos propensos a fallos o latencias (como servidores SMTP), se ejecutan de manera síncrona dentro de la solicitud de voz, degradarían gravemente la latencia percibida del asistente, bloqueando al usuario hasta que la conexión de red externa finalice.

## Decisión
Utilizaremos un patrón de **cola de mensajería asíncrona basada en ficheros JSON** para desacoplar las tareas de envío de notificaciones.

El flujo operará de la siguiente manera:
1. El servicio `orchestrator` se limita a generar la solicitud de notificación formateada en un payload JSON con un identificador de mensaje único.
2. El archivo JSON se escribe directamente en un volumen compartido de disco destinado a la bandeja de salida (`data/mail/pending/` en el host, montado como `/shared/mail/pending` en los contenedores). Esto finaliza la responsabilidad síncrona del orquestador.
3. Un microservicio independiente, desacoplado y especializado en el transporte (`mail-watchdog`) monitoriza el directorio `pending/`.
4. El `mail-watchdog` procesa los archivos, realiza la negociación de sockets de red y el envío SMTP, y elimina el archivo JSON tras una confirmación exitosa de envío.
5. En caso de fallo de red, se implementa una política de reintentos con backoff exponencial. Si los reintentos se agotan, el mensaje se traslada a la carpeta de fallos (`data/mail/failed/` en el host, correspondiente a `/shared/mail/failed` en el contenedor) para preservarlo de forma segura sin bloquear la cola de envío.

## Alternativas consideradas
- **Integración SMTP síncrona en el Orquestador:** Configurar las credenciales SMTP y el cliente de envío directamente en el microservicio `orchestrator`. Descartado por introducir una latencia de varios segundos en la respuesta de voz y carecer de resiliencia ante caídas del servidor SMTP (lo que provocaría la pérdida inmediata de la notificación).
- **Cola de mensajería en memoria (hilos secundarios en Python):** Procesar el envío en hilos paralelos dentro del orquestador. Descartado porque el `orchestrator` debe ser estrictamente sin estado (*stateless*); almacenar mensajes pendientes en memoria causaría la pérdida de datos ante reinicios del contenedor Docker.
- **Uso de un broker de mensajería clásico (como Redis con Celery / BullMQ):** Descartado por añadir dependencias operativas pesadas e infraestructura de red adicional innecesaria en el ecosistema local del asistente.

## Consecuencias
+ **Latencia de voz intacta:** El usuario recibe su respuesta por los altavoces de manera instantánea, sin esperar al envío físico del correo electrónico.
+ **Tolerancia a fallos y persistencia:** Los correos pendientes sobreviven a cortes de luz o reinicios del host al estar guardados físicamente en disco.
+ **Seguridad de datos:** Las credenciales del servidor SMTP se aíslan en el contenedor `mail-watchdog`, de forma que el `orchestrator` no tiene visibilidad de las contraseñas de correo del usuario.
- **Latencia de entrega:** El envío físico puede retrasarse unos segundos según el intervalo de polling y las re-evaluaciones del watchdog.
- **Posibilidad teórica de duplicados:** Si el servidor SMTP confirma el envío pero el daemon sufre una caída eléctrica antes de eliminar el archivo JSON de disco, el correo podría enviarse nuevamente al reiniciar el servicio (mitigado mediante identificadores únicos de transacción).
