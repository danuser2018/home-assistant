# ADR-004: Estandarización de APIs REST en el Ecosistema

## Fecha
28-06-2026

## Estado
Aceptado

## Contexto
El ecosistema de NOVA-2 está compuesto por múltiples microservicios desarrollados de forma independiente. Para que cooperen de manera síncrona mediante HTTP (como las llamadas `interaction-manager` → `stt-capability` / `orchestrator` / `tts-capability`), es crucial contar con una interfaz consistente. La falta de un estándar común de APIs provoca inconsistencia en el nombrado de endpoints, variabilidad en la serialización de datos y payloads de error dispares, lo que dificulta la integración y la depuración de problemas.

## Decisión
Establecemos un conjunto de reglas transversales de obligado cumplimiento para el diseño de APIs REST en todo el ecosistema:

1. **Versionado Explícito:** Todos los endpoints HTTP deben incluir la versión de la API en el prefijo de su ruta (por ejemplo, `/api/v1/` o `/v1/`). Está prohibido publicar endpoints productivos directamente en la raíz sin versionar.
2. **Esquema de Error Común:** Todas las respuestas con código de estado HTTP `4xx` o `5xx` deben devolver un payload JSON estructurado bajo el mismo esquema, indicando el tipo de error, un mensaje legible y detalles adicionales de forma consistente (inspirado en el estándar RFC 7807):
   ```json
   {
     "error": "NombreDelError",
     "message": "Mensaje descriptivo y legible para depuración",
     "status": 400
   }
   ```
3. **Nomenclatura Homogénea:** Queda prohibido mezclar estilos de nombrado en una misma interfaz. Se utilizarán recursos en minúsculas y se mantendrá una serialización homogénea en los campos JSON.
4. **Documentación de Contratos:** Cada servicio debe mantener actualizada su especificación OpenAPI (Swagger) y/o documentar con precisión sus contratos de entrada y salida en su correspondiente `README.md`.

## Alternativas consideradas
- **Estilo Libre Ad-hoc:** Permitir que cada microservicio implemente las APIs a criterio del desarrollador. Descartado porque incrementa la fricción al integrar nuevos componentes y obliga a añadir lógica condicional en el cliente (`interaction-manager`) para interpretar los diferentes formatos de respuesta.
- **Migración a gRPC/Protobuf:** Utilizar llamadas a procedimientos remotos de alto rendimiento en lugar de REST. Descartado porque, aunque ofrece contratos estrictos y tipados, añade complejidad operacional al requerir compilación de buffers en múltiples lenguajes (Python, JS, etc.) y limita el uso de herramientas de depuración directa como `curl`.

## Consecuencias
+ **Consistencia e Integrabilidad:** Integrar un nuevo microservicio o consumir un endpoint se vuelve predecible y rápido.
+ **Tratamiento robusto de errores:** El coordinador (`interaction-manager`) puede procesar de forma uniforme cualquier excepción de los servicios secundarios, mostrando errores de red claros en los logs.
+ **Retrocompatibilidad:** El versionado explícito protege a los consumidores de cambios disruptivos en la firma de los métodos.
- **Boilerplate adicional:** Requiere declarar middleware o manejadores globales de excepciones en cada framework para dar formato a los errores según el esquema común.
