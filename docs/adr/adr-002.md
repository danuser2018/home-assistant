# ADR-002: Modularización de Servicios (Host para Hardware y Docker para Procesamiento)

## Fecha
28-06-2026

## Estado
Aceptado

## Contexto
El asistente de voz local requiere interactuar con el hardware de sonido físico (micrófonos y altavoces) del host Linux de forma concurrente con el resto de aplicaciones de usuario. Los servicios de audio necesitan comunicarse de forma fiable con los servidores de sonido del usuario (PulseAudio o PipeWire), los cuales residen en la sesión de usuario activa de Linux. Por otro lado, la lógica de inteligencia artificial (modelos neuronales de traducción STT y síntesis TTS) y orquestación son componentes lógicos pesados que requieren entornos reproducibles y dependencias aisladas.

## Decisión
Dividiremos el ecosistema de servicios en dos planos de ejecución bien delimitados:

1. **Plano de Hardware (Host Nativo):** Los servicios de captura (`mic-daemon`) y reproducción de audio (`speaker-watchdog`) se implementarán y ejecutarán directamente sobre el sistema operativo del host como **servicios de usuario de systemd** (`systemd --user`). Esto les permite heredar y acceder de manera limpia al socket D-Bus de la sesión de audio del usuario sin configuraciones adicionales.
2. **Plano de Procesamiento (Docker):** El resto de servicios (`interaction-manager`, `stt-capability`, `orchestrator`, `tts-capability`, `system-service` y `mail-watchdog`) se empaquetarán y desplegarán en contenedores Docker aislados, comunicados a través de una red interna privada de Docker (`assistant-network`).

## Alternativas consideradas
- **Dockerización completa (incluyendo hardware de audio):** Intentar pasar sockets de PulseAudio/PipeWire y dispositivos ALSA `/dev/snd` dentro de los contenedores Docker. Descartado por ser extremadamente inestable, propenso a fallar ante actualizaciones del sistema operativo y muy difícil de configurar de forma portable para distintos usuarios y distribuciones Linux.
- **Despliegue nativo completo en el host:** Instalar todos los servicios y sus dependencias de IA directamente en el sistema del usuario. Descartado porque la instalación de librerías como CUDA, C++ compiler bindings, y versiones específicas de Python para los modelos neuronales rompería el objetivo de instalación en menos de 10 minutos y ensuciaría el sistema del usuario.

## Consecuencias
+ **Portabilidad acústica:** Acceso instantáneo y transparente al micrófono y altavoz seleccionados por el usuario en su escritorio (GNOME, KDE, etc.).
+ **Simplicidad de instalación:** Las pesadas librerías de IA se instalan en segundos descargando las imágenes precompiladas de Docker, sin conflicto de dependencias locales.
+ **Seguridad:** Los contenedores Docker no tienen privilegios ni visibilidad sobre los periféricos de entrada y salida de audio física del host.
- **Gestión mixta de servicios:** El usuario debe gestionar el ciclo de vida del sistema mediante dos herramientas diferentes (`systemctl --user` para audio y `docker compose` para procesamiento).
- **Consumo de recursos:** Sobrecarga ligera en memoria RAM y disco debido al aislamiento de contenedores Docker.
